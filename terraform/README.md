AARNN Network – Terraform Infrastructure and Local Containers

Overview
- This Terraform project builds and runs two containerized applications with selectable targets: Docker, Podman, or Kubernetes.
  - Aeron (github.com/aeron-io/aeron) – built from source, packaged with a Java runtime that starts the Media Driver by default.
  - AARNN (github.com/neuralmimicry/aarnn) – cloned and built into a C++-based container (CMake/Make), with a fallback health endpoint on port 8080.
- Optionally, the project can provision container registries in AWS, GCP, and Azure to host images. Pushing images is documented and intentionally not automatic for safety.

Key Features
- Reproducible builds via multi-stage Dockerfiles generated by Terraform templates.
- Robustness: long build timeouts, health checks, restart policies, isolated Docker network.
- Built-in observability: Prometheus and Grafana for metrics (Docker/Podman and Kubernetes). Optional logging with Loki + Promtail.
- Portability: local-only by default; clouds are opt-in using feature flags.
- Security-conscious: does not store secrets in state, avoids auto-push to registries, supports passing registry creds via variables.

Repository Layout
- terraform/
  - main.tf – orchestrates local builds and optional cloud registries
  - variables.tf, providers.tf, outputs.tf
  - modules/
    - local-apps/ – builds Docker images and starts containers
      - templates/Dockerfile.aeron.tftpl
      - templates/Dockerfile.aarnn.tftpl
    - aws-ecr/ – creates ECR repositories (no push)
    - gcp-artifact/ – creates Artifact Registry (no push)
    - azure-acr/ – creates Azure Container Registry (no push)
  - examples/
    - local-only.tfvars – example variables to run locally

Prerequisites
- Terraform >= 1.5
- Docker Engine with enough CPU/RAM and at least ~5 GB free disk space
- Internet access to clone GitHub repositories and pull base images
- If enabling cloud registry modules:
  - AWS CLI configured with credentials and a default account (for ECR)
  - gcloud authenticated with the target project (for Artifact Registry)
  - Azure CLI authenticated and a subscription selected (for ACR)

Quick Start
- Choose a deployment target via variable deployment_target: docker (default), podman, or kubernetes.

Docker (default)
1) cd terraform
2) terraform init
3) terraform apply -var-file=examples/local-only.tfvars
4) After apply: Containers <project_name>-aeron and <project_name>-aarnn running, network <project_name>-net, health at http://localhost:8080/health

Podman
1) Ensure Podman socket is available (rootless example): systemctl --user enable --now podman.socket
2) Export DOCKER_HOST to Podman socket, e.g.: export DOCKER_HOST=unix:///run/user/$(id -u)/podman/podman.sock
3) Optionally force runtime detection: export TF_VAR_container_runtime=podman
4) cd terraform && terraform init
5) terraform apply -var-file=examples/podman-local.tfvars

Kubernetes (kind/minikube or any kubeconfig)
1) Have a cluster and kubeconfig ready (e.g., kind create cluster --name aarnn)
2) Set kubeconfig context to your cluster. For kind, it is usually kind-aarnn. Either export TF_VAR_kubeconfig_context=kind-aarnn, or edit terraform/examples/k8s-kind.tfvars and set kubeconfig_context = "kind-aarnn".
3) cd terraform && terraform init
4) terraform apply -var-file=examples/k8s-kind.tfvars
5) If using local images and a local cluster, load images into kind (before or after apply):
   kind load docker-image aeron-local:latest --name aarnn
   kind load docker-image aarnn-local:latest --name aarnn
6) Access AARNN via Service <project_name>-aarnn in namespace k8s_namespace (ClusterIP by default). For NodePort/LoadBalancer, set k8s_service_type.

Auto-loading images into kind:
- When deployment_target = kubernetes and kubeconfig_context starts with "kind-", Terraform will attempt to run `kind load docker-image aeron-local:latest` and `aarnn-local:latest` for the detected kind cluster name, if no image overrides are set.
- Requirements: the images must exist in your local Docker daemon (kind uses Docker). If you built with Podman, rebuild images with Docker or push to a registry and set the image override vars.

Build images for Kubernetes (no registry):
- Set build_k8s_local_images = true to have Terraform build Aeron and AARNN images locally even when deployment_target = kubernetes, and then auto-load them into kind (when kubeconfig_context starts with kind-).
- This enables a fully local workflow without pushing to a registry. Note: kind uses the Docker daemon; ensure Docker is available (not Podman) for this step.
- Alternatively, push images to a registry and set aeron_image_override / aarnn_image_override.

Troubleshooting (Kubernetes):
- Error like "no route to host" or API server 192.168.x.x unreachable usually indicates kubectl/terraform are pointing at the wrong context/cluster. Run kubectl config get-contexts and set TF_VAR_kubeconfig_context to the intended one (e.g., kind-aarnn), or update kubeconfig_context in your tfvars.
- Error: Provider configuration: cannot load Kubernetes client config / context "kind-aarnn" does not exist
  - Cause: the kind cluster was created as root (using sudo), so its kubeconfig context was written to /root/.kube/config and is invisible to your user’s ~/.kube/config.
  - Fix option A (recommended): export the kind kubeconfig into your user config:
    kind export kubeconfig --name aarnn --kubeconfig ~/.kube/config
    Then ensure kubeconfig_context = "kind-aarnn" (see examples/k8s-kind.tfvars) or run: kubectl config use-context kind-aarnn
  - Fix option B: explicitly point Terraform at the kubeconfig that actually contains the context:
    - Set TF_VAR_kubeconfig_path=/root/.kube/config (if running terraform with sudo -E) or
    - In your tfvars, set kubeconfig_path = "/root/.kube/config" (only when appropriate). Avoid mixing sudo and non-sudo between kind and terraform.
- General tip: avoid sudo for kind and Terraform. Prefer rootless Docker/Podman and kind so contexts land in ~/.kube/config.

Container Details
- Aeron container
  - Image built from the Aeron source at aeron_git_ref (default master)
  - Default command runs the Media Driver
  - Health check ensures a Java process is running
- AARNN container
  - Image built from the AARNN source at aarnn_git_ref (default main)
  - Exposes port 8080 and provides a fallback /health endpoint
  - Health check uses curl against localhost:8080/health

Feature Flags
- enable_local_containers (bool, default true): Build and run local containers
- enable_aws_ecr (bool, default false): Create AWS ECR repos
- enable_gcp_artifact (bool, default false): Create GCP Artifact Registry repo
- enable_azure_acr (bool, default false): Create Azure Container Registry

Cloud Registries (Manual Push Guide)
After building locally, you may tag and push images to your cloud registries. The modules output useful values.

AWS ECR
- Enable the module and apply:
  terraform apply -var enable_aws_ecr=true
- Get outputs:
  terraform output aws_ecr_urls
- Authenticate and push (example):
  aws ecr get-login-password --region <region> | docker login --username AWS --password-stdin <account>.dkr.ecr.<region>.amazonaws.com
  docker tag aeron-local:latest <repo_url>:aeron
  docker push <repo_url>:aeron
  docker tag aarnn-local:latest <repo_url>:aarnn
  docker push <repo_url>:aarnn

GCP Artifact Registry
- Enable the module and apply:
  terraform apply -var enable_gcp_artifact=true -var gcp_project=<project> -var gcp_region=<region>
- Create Docker auth helper and login:
  gcloud auth configure-docker <region>-docker.pkg.dev
- Tag and push:
  docker tag aeron-local:latest <region>-docker.pkg.dev/<project>/<repo>/aeron:latest
  docker push <region>-docker.pkg.dev/<project>/<repo>/aeron:latest
  docker tag aarnn-local:latest <region>-docker.pkg.dev/<project>/<repo>/aarnn:latest
  docker push <region>-docker.pkg.dev/<project>/<repo>/aarnn:latest

Azure ACR
- Enable the module and apply:
  terraform apply -var enable_azure_acr=true -var azure_location=<loc> [-var azure_resource_group=<rg>]
- Get login server:
  az acr login --name <acrName>  # or docker login <login_server>
- Tag and push:
  docker tag aeron-local:latest <login_server>/aeron:latest
  docker push <login_server>/aeron:latest
  docker tag aarnn-local:latest <login_server>/aarnn:latest
  docker push <login_server>/aarnn:latest

Variables and configuration (complete)
This project exposes variables grouped by area. You can set them via:
- A tfvars file: terraform apply -var-file=examples/local-only.tfvars
- CLI -var flags: terraform apply -var deployment_target=kubernetes
- Environment variables: export TF_VAR_kubeconfig_path=~/.kube/config

Global
- project_name (string, default "aarnn-net"): Namespacing for resources (container names, network, K8s names).
- default_tags (map(string), default {project=aarnn-network, managed=terraform}): Applied where supported.
- manifest_path (string, default null): Path to apps manifest YAML. Defaults to terraform/apps.manifest.yaml.

Deployment target and features
- deployment_target (string, docker|podman|kubernetes; default docker): Where to run.
- enable_local_containers (bool, default true): Build and run local containers for Docker/Podman targets.
- enable_monitoring (bool, default true): Enable Prometheus/Grafana.
- enable_logging (bool, default false): Enable Loki/Promtail (local) or loki-stack (K8s).

Docker/Podman
- docker_host (string, default null): Docker-compatible socket (set for Podman). Example: unix:///run/user/1000/podman/podman.sock
- docker_registry_address (string, default null): Registry hostname (e.g., index.docker.io, <acct>.dkr.ecr.<region>.amazonaws.com)
- docker_registry_username (string, default null, sensitive): Registry username.
- docker_registry_password (string, default null, sensitive): Registry password/token.

Kubernetes
- kubeconfig_path (string, default null): Path to kubeconfig (defaults to ~/.kube/config).
- kubeconfig_context (string, default null): Optional context name.
- k8s_namespace (string, default "aarnn"): Namespace for apps and monitoring.
- k8s_service_type (string, ClusterIP|NodePort|LoadBalancer; default ClusterIP): Service type for AARNN and extras.
- aeron_image_override (string, default null): Override image for Aeron.
- aarnn_image_override (string, default null): Override image for AARNN.
- node_selector (map(string), default {}): Scheduling hints (e.g., {"kubernetes.io/arch"="arm64"}).
- tolerations (list(object), default []): Pod tolerations.

Monitoring and logging
- grafana_admin_user (string, default "admin"): Grafana admin username.
- grafana_admin_password (string, default null, sensitive): Grafana admin password (defaults to "admin" when null; dev only).

GPU and architecture
- enable_gpu (bool, default false): Request/allow GPUs for Docker/Podman and Kubernetes.
- gpu_count (number, default 1): GPUs to request (Docker device requests; Kubernetes limits nvidia.com/gpu).
- target_arch (string, amd64|arm64|null; default null): Local build platform (linux/<arch>). Null uses host.

Cloud registries (optional – resources only, no automatic push)
- enable_aws_ecr (bool, default false), aws_region (string, default "us-east-1").
- enable_gcp_artifact (bool, default false), gcp_project (string, default null), gcp_region (string, default "us-central1").
- enable_azure_acr (bool, default false), azure_location (string, default "eastus"), azure_resource_group (string, default null).

Harbor registry on Kubernetes (useful for OpenStack/on‑prem)
- enable_harbor (bool, default false): Install Harbor via Helm into your cluster.
- harbor_expose_ingress (bool, default false): Expose via Ingress (requires an ingress controller and DNS).
- harbor_hostname (string, default null): Hostname to use when exposing via Ingress.
- harbor_storage_class (string, default null): StorageClass for Harbor persistence.

App build and runtime sizing (local Docker/Podman)
- aeron_git_ref (string, default "master").
- aarnn_git_ref (string, default "main").
- aeron_container_cpu (number, default 1024), aeron_container_memory (MB, default 1024).
- aarnn_container_cpu (number, default 1024), aarnn_container_memory (MB, default 2048).

Manifest-driven generic apps
- apps.manifest.yaml defines additional apps. Generic apps can specify: name, repo, ref, type=generic, port, env, cmd, health_path. For Kubernetes, generic apps deploy when an image field is provided.

Sensitive variables
- docker_registry_username, docker_registry_password, grafana_admin_password are marked sensitive and hidden from terraform output by default.

Credentials and secrets
Provide credentials via your tooling/CLI wherever possible. Avoid putting secrets into tfvars files or committing them to VCS.

Docker/Podman registry credentials
- Preferred: perform docker login to your registry before apply.
- Alternative: provide docker_registry_username and docker_registry_password (sensitive). Example:
  export TF_VAR_docker_registry_address=index.docker.io
  export TF_VAR_docker_registry_username=myuser
  export TF_VAR_docker_registry_password=$(pass show docker/token)

AWS (ECR)
- Use the AWS CLI/SDK credentials chain (env vars or ~/.aws). Example:
  export AWS_REGION=us-east-1
  export AWS_PROFILE=dev
- The aws-ecr module creates repositories; push manually as documented below.

GCP (Artifact Registry)
- Authenticate with gcloud to the target project: gcloud auth login; gcloud config set project <id>.
- For pushing images: gcloud auth configure-docker <region>-docker.pkg.dev

Azure (ACR)
- Authenticate with Azure CLI: az login; az account set --subscription <id>.
- az acr login --name <acrName> to push.

Kubernetes
- Access via kubeconfig_path/context. Ensure your kubeconfig has credentials for the cluster. Example:
  export TF_VAR_kubeconfig_path=~/.kube/config

OpenStack
- This project does not require OpenStack credentials by default. Harbor installs into any existing Kubernetes cluster via Helm. If you deploy Kubernetes on OpenStack (e.g., Magnum), use your standard kubeconfig.

Git access for private repositories
- The provided Dockerfiles clone public Git repos. Private repos are not supported out-of-the-box. If needed, extend the Dockerfiles to inject SSH keys or tokens via build args and secret mounts (buildx) and never commit secrets to VCS.

Security notes about state and secrets
- Sensitive variables are kept out of plan output but can still reside in Terraform state if used by resources. Do not commit terraform.tfstate. Prefer passing secrets via environment variables, CI secret stores, or a local.tfvars not checked into version control.

Security Notes
- No credentials are hard-coded. Provide any registry or cloud credentials via environment/CLI where possible.
- Terraform state may contain resource names and metadata; avoid storing secrets in variables or state.

Troubleshooting
- Docker build fails for Aeron:
  - Ensure enough RAM/CPU; the Java build can be heavy. Increase memory to Docker if using Docker Desktop.
  - The Dockerfile tries Gradle first then falls back to Maven. Check logs within the build output for the failing step.
- AARNN container exits immediately:
  - The project may not expose an entrypoint. The image falls back to a tiny health server; check logs: docker logs <project_name>-aarnn
- Port conflicts on 8080:
  - Edit the container port mapping in modules/local-apps/main.tf or override via a variable (consider enhancing module to parameterize port).
- Networking:
  - Both containers are attached to a dedicated network <project_name>-net. Use container names to resolve services.

Maintenance & Extensibility
- Terraform code is modular; cloud registry support is separated.
- Single manifest to manage apps: terraform/apps.manifest.yaml
  - This file is the source of truth for what to clone/build/deploy.
  - Default entries cover Aeron and AARNN. Add more under `apps` with fields:
    - name: short identifier (used in image/container names)
    - repo: git clone URL
    - ref: branch/tag/commit
    - type: aeron | aarnn | generic
    - port: optional container port to expose locally
    - cmd: optional command array to run in container
    - env: optional map of ENV vars
    - health_path: optional path for healthchecks (defaults to /health if port is set)
  - Example extra app:
    - name: mysvc
      repo: https://github.com/example/my-python-svc.git
      ref: main
      type: generic
      port: 8081
      cmd: ["bash","-lc","python -m mysvc"]
      env: { LOG_LEVEL: "info" }
  - Override manifest path via -var manifest_path=/abs/path/apps.yaml
  - Notes:
    - Today, extra generic apps are built and run for Docker/Podman targets and can also be deployed to Kubernetes when an image is specified in the manifest entry (field `image`).
- To change build logic, update the Dockerfile templates under modules/local-apps/templates.
- To add environment variables or volumes, edit the docker_container resources in modules/local-apps/main.tf.
